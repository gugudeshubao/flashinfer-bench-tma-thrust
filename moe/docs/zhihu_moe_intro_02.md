# MoE 算子怎么从 reference 版，一步步优化到真正可用的 GPU 实现

上一篇我讲的是：

**MoE 算子本质上到底在做什么，为什么它实现起来比普通 FFN 难。**

这一篇我想继续讲一个更实际的问题：

> 如果我真的要把 MoE 算子写出来，应该按什么顺序做，才能又快又稳地走到一个真正可用的 GPU 实现？

这篇文章我不会讲太多“理想化设计”，而是直接从我自己的实现和测试经验出发，讲我会怎么推进一版 MoE 算子。

核心观点先放在前面：

> 我不会一上来就追求“全融合、全自定义、一步到位”，我会先把正确性建立起来，再把不规则路由问题收敛，最后才逐步把热点段落下沉到 Triton / CUDA / CuTe / PTX。

这不是保守，而是工程上最省时间的路线。

---

## 1. 第一阶段：先写一个绝对正确的 reference

我写 MoE 的第一步，永远不是优化，而是先写一个慢但确定正确的版本。

这个版本的目标只有一个：

- 以后所有优化版的输出，都拿它来对齐

最朴素的 reference 大概长这样：

```python
def moe_ref(x, gate_w, w1, w2, topk):
    logits = x @ gate_w                       # [T, E]
    topk_vals, topk_ids = torch.topk(logits, topk, dim=-1)
    topk_weights = torch.softmax(topk_vals, dim=-1)

    y = torch.zeros_like(x)
    for t in range(x.shape[0]):
        for j in range(topk):
            e = topk_ids[t, j].item()
            w = topk_weights[t, j]
            h = x[t] @ w1[e]
            h = act(h)
            o = h @ w2[e]
            y[t] += w * o
    return y
```

这个版本当然很慢，但我非常看重它，因为后面无论我做：

- token 重排
- grouped GEMM
- Triton kernel
- CUDA/C++ 扩展
- CuTe/C++ 实现

我都需要一个“地面真值”。

我自己的经验是，**没有 reference，就没有优化，只有混乱。**

---

## 2. 第二阶段：先把逻辑从“逐 token”变成“按 expert 分桶”

当我确认 reference 没问题后，我不会立刻去写 fused kernel。

我会先做一版**按 expert 分桶**的实现。

因为 MoE 真正的结构性问题是：

- token 只去少量 expert
- 每个 expert 收到的 token 数不同
- 数据访问不规则

所以我一定会先把路由结果整理成这种形态：

- `flat_token_ids [T*K]`
- `flat_expert_ids [T*K]`
- `flat_weights [T*K]`

然后做：

1. histogram
2. prefix sum
3. counting-sort / bucket
4. gather expert input

最后得到：

- 同一个 expert 的 token 在内存里连续

这样 expert FFN 至少可以按连续批处理输入来算。

这是我认为第一个真正“可用”的版本。

它还不一定快，但至少结构已经正确了。

---

## 3. 第三阶段：接受一个现实，瓶颈往往不是 GEMM

很多人第一次写 MoE，会默认觉得：

> 性能瓶颈一定在 expert FFN 的两个 GEMM。

但我实际做下来，尤其在推理、小 batch、短序列下，往往不是这样。

我更常见到的瓶颈其实是：

- top-k 和路由元数据生成
- token 重排 / gather / scatter
- 小 expert batch 导致的 launch overhead
- Python 层调度过多

也就是说，MoE 不是单纯的大矩阵乘法问题，而是**不规则数据流问题**。

所以我会尽量用下面这个顺序思考优化：

1. 先减少不必要的中间张量
2. 再减少 launch 次数
3. 再看有没有必要把热点段落下沉到低层

如果一开始就盯着 GEMM，很容易把力气花错地方。

---

## 4. 第四阶段：我怎么做性能验证

这是我觉得最重要，但也最容易被忽视的一点。

我不会每改一点就直接跑完整 benchmark。

我会把验证拆成两个层次：

### 4.1 小 workload smoke test

比如只跑 1 到 2 个 workload，快速回答两个问题：

- 这个版本能不能编译 / 运行
- 在最小场景下有没有明显收益或明显退化

这种测试非常适合验证：

- 新的 Triton kernel
- 新的 CUDA 扩展
- 新的路由逻辑
- 新的缓存策略

### 4.2 全 workload sweep

只有当小 smoke test 通过后，我才会做完整 sweep。

因为很多优化只在某些规模上有效。

我自己实际测下来，非常典型的情况就是：

- 小 workload 上提升明显
- 中等 workload 上持平
- 大 workload 上反而退化

如果没有完整 sweep，很容易被局部最优骗了。

---

## 5. 第五阶段：我如何判断一个优化值不值得继续

这部分我想讲得更直接一点。

我不会因为一个优化“看起来很高级”就继续做。

我只看三件事：

### 5.1 它是不是稳定正确

如果一个版本：

- 需要很多特判
- 容易在不同 workload 上炸
- 会出现奇怪的 shape / runtime 错误

那即使它偶尔快一点，我也不会把它当主线。

### 5.2 它是不是有全局收益

我不会被“某个 workload 上 1.2x 提升”冲昏头。

我只关心：

- 它在完整 workload 集合上的平均结果如何

因为比赛也好，真实系统也好，最后都不会只跑一个点。

### 5.3 它会不会显著增加复杂度

有些优化收益很小，但复杂度暴涨。

这种我通常会直接放弃。

比如：

- 很复杂的缓存逻辑
- 很脆弱的 hybrid 分支
- 看似优雅但很难 debug 的全融合实现

我越来越相信一句话：

> 真正好的优化，不只是快，还要可控、可验证、可维护。

---

## 6. 我踩过的几个典型坑

我觉得这一部分对刚开始写 MoE 的人最有帮助。

### 6.1 以为低层实现一定更快

不一定。

我实际测过很多版本，结论非常明确：

- 有些 PTX / CuTe / CUDA 方案只在小 workload 上更快
- 一到中大 workload，优势就没了
- 有些版本甚至会因为额外的准备开销反而更慢

### 6.2 过早做缓存

我一开始也很容易觉得：

“既然 correctness 和 benchmark 会重复调用，那我把各种中间结果缓存起来肯定更快。”

但现实是：

- 一些缓存会引入错误命中
- 一些缓存本身的维护成本就不低
- 有些缓存会明显拉高小 workload 的固定开销

所以我现在只愿意保留两类缓存：

- 明确精确
- 规模小、风险低

### 6.3 过早做 hybrid

比如：

- 小输入走 A
- 大输入走 B
- 某些 expert 数量走 C

这种策略短期看起来很聪明，但后面调试和维护成本会迅速膨胀。

所以我现在通常会先把单一路线跑通，再考虑 hybrid。

---

## 7. 我现在最推荐的实现顺序

如果我今天重新从零写一版 MoE forward，我会严格按这个顺序走：

### 第一步：Reference

先有一个慢但绝对正确的版本。

### 第二步：按 expert 分桶

把 dispatch / gather / combine 逻辑做清楚。

### 第三步：稳定基线

做一版结构简单、全 workload 不退化的 GPU 实现。

### 第四步：低层热点下沉

优先下沉这些段落：

- top-k / route metadata
- token 重排
- combine
- activation / postprocess

### 第五步：完整 sweep

只有完整 benchmark 过了，我才会把这个版本视为真正可用。

---

## 8. 结合我自己的实践，什么样的优化最值得做

如果只讲“第一次往可用实现推进”，我现在最认可的方向是：

### 8.1 先把不规则路由收敛

先把：

- top-k
- expert 分桶
- token gather / scatter

这些做稳定。

### 8.2 再减少 launch overhead

很多时候，真正值钱的是把一些很碎的小算子合在一起。

### 8.3 再决定是否下沉到 CUDA/C++ / CuTe

低层方案值得做，但一定是建立在：

- 已经知道瓶颈在哪
- 已经知道它在小 smoke test 里有收益
- 已经知道它不会全局退化

的前提下。

---

## 9. 我对“MoE 优化”这件事的最终理解

写到这里，我最想强调的一句话其实是：

> MoE 优化不是“把 FFN 写得更花哨”，而是“把稀疏、不规则、动态路由这件事收拾干净”。

这才是 MoE 算子真正的工程难点。

如果这件事没有做好，那么后面无论是 Triton、CUDA、PTX、CuTe，都会变成在混乱之上叠复杂度。

而如果这件事做好了，那么后面的低层优化才有意义。

---

## 10. 这篇文章的结尾，我想给出一个更实际的建议

如果你现在也在写 MoE 算子，我会建议你先别问：

- 我该不该上 CuTe
- 我该不该写 PTX
- 我该不该做全融合

而是先问：

- 我的 route metadata 对了吗
- 我的 token 分桶连续了吗
- 我的 combine 逻辑稳定了吗
- 我的小 smoke test 和完整 sweep 是不是都过了

这些问题答清楚以后，优化路线会自然很多。
