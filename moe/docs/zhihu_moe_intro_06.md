# MoE 算子到底该怎么做 benchmark？为什么很多优化小 workload 上成立，完整 sweep 里却会退化

前面几篇我已经把 MoE 前向的核心结构基本拆开讲完了：

1. MoE 算子本质是什么  
2. 我怎么从 reference 版走到可用 GPU 实现  
3. top-k、counting sort、token 重排  
4. expert FFN 的 grouped / batched 路线  
5. combine / scatter-add 的真正难点  

这一篇我想讲一个我自己做 MoE 优化时越来越重视的话题：

> MoE 算子到底该怎么 benchmark，为什么很多优化在小 workload 上成立，但一到完整 sweep 就开始退化？

我觉得这是 MoE 里一个特别容易被低估的工程问题。

因为如果 benchmark 没做对，后面很多“优化结论”其实都是幻觉。

---

## 1. 为什么我会越来越重视 benchmark

最开始写算子的时候，我也很容易有一种冲动：

- 改一点
- 跑一下
- 如果更快，就觉得自己找到了方向

但 MoE 不是一个那么容易被单点 benchmark 概括的问题。

它有几个非常麻烦的特点：

- token 数变化大
- expert 分布动态变化
- 不同 workload 下 `N_e` 差异很大
- 小 workload 和中大 workload 的瓶颈可能完全不同

也就是说，MoE 很少存在“一个版本在所有规模上都更优”这种简单结论。

我后来越来越相信一句话：

> 如果我只看一个 workload 上的结果，我通常只是在看一个局部现象，而不是在看一个实现的真实质量。

---

## 2. 为什么 MoE 特别容易出现“局部最优”

普通 dense GEMM 往往比较简单：

- 输入 shape 规则
- 行为比较稳定
- 性能趋势通常也比较一致

MoE 完全不是这样。

同一个实现，可能在不同 workload 上表现差很多。

我实际做 MoE 时，最常见到的就是这种情况：

### 2.1 小 workload 上更快

某个低层优化版本因为：

- launch overhead 更小
- 激活 kernel 更紧
- Python 调度更少

所以小 workload 上会明显更快。

### 2.2 中段 workload 上持平

一旦 token 数和 expert batch 稍微大一点，

- 这些小固定开销的收益就被摊薄了

于是它和基线开始接近。

### 2.3 大 workload 上反而更慢

再往上走，问题又变了：

- 额外缓存开始拖后腿
- 路由重排变得更重
- 额外的中间张量占用变多
- 原本为了小场景写的特判反而增加了复杂度

于是它又掉回去了。

这就是 MoE 最典型的 benchmark 陷阱：

> 一个版本在局部 workload 上看起来非常漂亮，并不代表它是好的主线实现。

---

## 3. 我现在怎么定义“一个 benchmark 做得合格”

如果我现在要判断一个 MoE 实现到底值不值得留，我至少会看三层结果。

### 第一层：Reference correctness

先不谈性能，只看：

- shape 对不对
- dtype 对不对
- 数值误差在不在容忍范围内

我几乎不会跳过这一层。

因为很多“快了”的版本，本质上只是把结果算错了。

### 第二层：小规模 smoke test

我通常会先选非常少的 workload，比如：

- 1 个 workload
- 或前 2 个 workload

它的作用不是证明版本好，而是快速回答两个问题：

1. 这个版本能不能编译 / 运行 / 通过 correctness  
2. 在最敏感的小场景下，它有没有明显收益或明显退化

这一层对我非常重要，因为它决定我值不值得继续把这个版本往下推。

### 第三层：完整 workload sweep

只有当 smoke test 过了，我才会做完整 sweep。

因为真正让我信服的，不是：

- “它在 workload A 上快了 20%”

而是：

- “它在整个 workload 集合上的平均表现更好，而且没有明显坏点”

这才是一个版本有没有资格进主线的标准。

---

## 4. 为什么我几乎不会直接上来跑全套 benchmark

这是一个看起来有点“保守”，但实际上非常省时间的习惯。

如果我每改一点都直接跑全套：

- 成本很高
- 反馈很慢
- 很多明显有问题的版本也会浪费大量时间

而且更麻烦的是：

- 如果全套 benchmark 失败了，我反而很难快速定位问题

所以我现在更喜欢这个节奏：

1. 先 smoke  
2. 再局部对比  
3. 最后完整 sweep

这个节奏的好处是：

- 小问题早暴露
- 坏方向可以早点砍掉
- 真正值得留的版本才配得上完整 benchmark

---

## 5. 我为什么越来越相信“先小后大”的验证方式

因为很多问题只在某一类 workload 上最明显。

### 5.1 小 workload 最容易暴露固定开销

比如：

- launch overhead
- Python 调度
- 小 kernel 过碎
- 不必要的中间 tensor

这些问题在小 workload 下最明显。

所以如果一个版本在小 workload 上就明显更差，我通常不会再对它抱太大希望。

### 5.2 中段 workload 最容易暴露结构性问题

比如：

- dispatch 是否真的规整
- grouped GEMM 是否开始发挥作用
- combine 是否开始变成热点

中段 workload 往往能告诉我：

这个实现到底只是一个“小样本优化技巧”，还是结构上真的更合理。

### 5.3 大 workload 最容易暴露全局退化

比如：

- padding 浪费
- 额外缓存开销
- 多次重排
- 内存占用过大

很多版本就是在这里开始掉速的。

所以如果我只做小 benchmark，不看中大 workload，我通常是不放心把它推上主线的。

---

## 6. 我如何看待“平均 speedup”

我现在越来越不喜欢只看某个点的速度。

我更关心的是：

\[
\text{average speedup over workload set}
\]

原因很简单：

- 真正的系统不会只跑一个 shape
- 真正的比赛也不会只测一个点
- 一个实现如果只在某个点上很强，但整体不稳，那它不是好的主线

所以我在做版本筛选时，越来越像这样判断：

### 一个版本可以继续保留，如果：

- correctness 稳定
- 小 workload 不明显退化
- 中段 workload 至少不掉太多
- 完整 sweep 的平均值有优势

### 一个版本应该被砍掉，如果：

- correctness 不稳
- 只在局部 workload 漂亮
- 完整 sweep 平均没赢
- debug 和维护成本太高

这套标准对我来说，比看单点 speedup 更可靠得多。

---

## 7. 我踩过的 benchmark 坑

这一部分我觉得特别真实。

### 7.1 被小样本优势骗了

有些版本在前两个 workload 上非常漂亮，
但一做完整 sweep 就掉回去了。

这类情况我见过很多次。

### 7.2 低估了缓存污染问题

尤其是我自己做过很多运行时编译 / 扩展缓存的实验之后，更有体会：

- 旧编译产物
- 缓存键设计错误
- 跨 workload 误命中

这些问题都可能让 benchmark 结果变得不可信。

如果我不主动防这个问题，很容易误判一个版本的真实质量。

### 7.3 以为“通过 correctness”就等于稳定

不一定。

有些版本：

- correctness 阶段过了
- timing 阶段才炸

这通常意味着：

- 某些路径只在重复调用时触发
- 某些缓存逻辑有问题
- 某些中间状态没有被正确隔离

所以我现在会更重视：

> correctness 和 timing 都要一起看。

### 7.4 忽视了 benchmark 本身的结构

很多 benchmark 框架不是简单调用一次算子就结束。

它们常常是：

- 先 correctness
- 再 warmup
- 再 timing

这就意味着：

- 一些“只在第二次调用才生效”的优化会被放大
- 一些“只在第二次调用才出错”的 bug 也会被放大

如果我没理解 benchmark 本身的执行结构，就很容易得出错误结论。

---

## 8. 我现在如何设计 benchmark 节奏

如果我要推进一个新版本，我通常会这样做：

### 第一步：最小 smoke

只回答：

- 能不能跑
- 会不会炸
- 数值对不对

### 第二步：代表性 workload 对比

一般我会选：

- 一个小 workload
- 一个中段 workload

这个阶段的目的，是快速看“趋势”。

### 第三步：完整 sweep

只有当第二步看起来确实值得，我才会做全 workload benchmark。

这套节奏的核心不是节省时间而已，而是：

> 我不想让大规模 benchmark 成为一种“替代思考”的工具。

很多时候，大 benchmark 不是用来找方向的，而是用来确认方向的。

方向应该先在小验证里找出来。

---

## 9. 一个实现真正“值得上主线”，在我这里需要满足什么

我现在大概只看四件事：

### 9.1 它数值稳定

不是偶尔通过，而是稳定通过。

### 9.2 它不是只在局部 workload 漂亮

如果只是一个点很好看，我不会太激动。

### 9.3 它的平均值确实更好

这点很重要。

### 9.4 它没有把后续维护成本抬得太高

如果一个版本快了一点点，但调试和维护成本翻倍，我通常不会把它当主线。

我越来越相信：

> 主线版本首先应该是“稳且可控”，其次才是“局部极致”。

---

## 10. 如果让我总结这一篇，我最想留下的一句话

我现在对 MoE benchmark 的理解，可以总结成一句话：

> 对 MoE 来说，benchmark 不是“证明我写得多快”，而是“防止我被局部最优骗了”。

这就是为什么我越来越依赖：

- smoke test
- 代表性 workload
- 完整 sweep

这一整套节奏。

它不是形式主义，而是让我在复杂优化里保持清醒的工具。
