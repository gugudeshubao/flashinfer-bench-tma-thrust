# Gated DeltaNet 最大的误解：大家都在盯 FLOPs，但真正卡死它的是状态

如果你最近在看 Gated DeltaNet、Delta-rule、线性注意力这条路线，大概率会被一种说法打动：

它不像 softmax attention 那样有一整个 `L x L` 的 attention matrix，理论复杂度更线性，长序列更友好，看起来简直像是“下一代更优雅的 attention 替代品”。

但我这次真正把 GDN workload 放到 GPU 上跑，尤其结合比赛里的：

- `gdn_decode_qk4_v8_d128_k_last`
- `gdn_prefill_qk4_v8_d128_k_last`

再去看它的 `State Size`、`BW (GB/s)`、batch 变化下的带宽曲线，最后得出的结论反而很直接，甚至有点反直觉：

**GDN 最难的地方，根本不是 FLOPs。**

更准确地说：

**GDN 真正的系统瓶颈，是“状态递推难并行 + state 读写吃带宽 + 低精度还容易炸”三件事同时成立。**

这也是为什么很多人会觉得它“论文里很好看，工程上却没那么顺滑”。不是因为它没有理论价值，而是因为它优化的对象，压根就不是一类简单的矩阵乘问题。

如果只用一句话提前剧透全文，那就是：

**attention 的核心资源是矩阵乘，GDN 的核心资源是状态。**

而 GPU 恰恰天生更偏爱前者。

## 一、很多人一开始就看错了：GDN 不只是“另一种 attention”

标准 attention 的直觉很简单：

- 算 token 之间的相关性
- 做归一化
- 再聚合出输出

你可以嫌它复杂度高，可以嫌它显存占用大，但它有一个巨大的工程优势：

**它太容易被改写成矩阵问题了。**

矩阵乘是什么？是现代 GPU 的主场，是硬件、编译器、库生态投入最大的地方。只要你的问题还能不断往 GEMM 靠，通常就还有很多性能红利可以吃。

而 GDN 不是这样。

GDN / Delta-rule 这类结构的核心，不是构造一张 token-token 关系图，而是**沿着时间轴递推一个状态 `state`**。每来一个 token，都要做这几件事：

1. 读当前 token 的 `q, k, v`
2. 读 gate 参数，比如 `a / b / dt_bias / A_log`
3. 更新旧状态 `state`
4. 再从新状态里产生当前输出

注意，问题就从这里变了。

因为一旦模型的中心对象从“矩阵交互”变成“状态递推”，整个优化逻辑都会跟着变：

- 你不再天然拥有大块并行
- 你必须频繁读写状态
- 你必须关心递推过程的数值稳定

也就是说，GDN 根本不是把 attention 换个公式那么简单，它是在把问题从“矩阵系统”换成“状态系统”。

## 二、GDN 的第一刀，不是砍在算力上，而是砍在并行性上

GDN 最先暴露出来的问题，不是慢在算不动，而是慢在不好并行。

为什么？

因为它的 `state` 是递推的。

这意味着当前 token 的结果，依赖上一步的 `state`；下一步又依赖当前这一步刚更新完的 `state`。这和标准 attention 在 prefill 里那种“整段序列一起做矩阵计算”的感觉完全不同。

于是 decode 阶段会立刻出现一个硬约束：

**时间维并行性很弱。**

这会带来什么？

- 单 token latency 会非常关键
- kernel launch overhead 会被无限放大
- 小 batch 下 GPU 吃不满会很明显
- 很多 attention 里常见的“大块并行”手段根本不好使

这也是为什么我越来越觉得，GDN 的 decode 更像 RNN/SSM 风格的系统问题，而不是传统 attention 那种“算子够不够厚”的问题。

换句话说，GDN 的第一瓶颈不是“算太多”，而是：

**它太递推了。**

## 三、真正卡住 GDN 的，往往不是计算量，而是 state 在显存里来回搬

如果你只盯 FLOPs，很容易误判 GDN 的瓶颈。

因为在真实实现里，GDN 很多时候并不是 compute-bound，而是非常典型的 memory-bound。

这次 workload 的一个关键信号，就是 benchmark 里直接给了：

- `State Size`
- `BW (GB/s)`

这其实已经在明示了：这个算子的性能，很可能主要取决于你怎么搬 state，而不是你怎么堆更多算力。

比如典型状态张量就是：

```python
state = [B, 8, 128, 128]
```

别看这个 shape 好像不夸张，一旦进入 decode，它的代价会被无限放大。

每生成一个 token，你都要：

1. 把 state 从 HBM 或 cache 读出来
2. 做衰减、混合、外积更新
3. 再把新 state 写回去

而问题在于，这整一步的计算量并不厚。

对于 decode，一个 batch element 大致是：

- 8 个 head
- 总计算量约 `1.05M FLOPs`
- 总状态流量约 `1.05 MB`

所以算术强度几乎只有：

```text
AI ≈ 1 FLOP / byte
```

这是什么概念？

这意味着它离 compute roof 很远，更像是一个注定先被带宽卡住的问题。

也就是说，对 GDN decode 来说，很多时候你不是在和“峰值算力”竞争，而是在和“每一步到底搬了多少字节、搬得顺不顺”竞争。

所以我会把这件事说得更直接一点：

**GDN decode 最大的敌人，不是算力不够，而是 state 太重。**

## 四、最容易被低估的第三刀：数值稳定性会直接决定你还能不能继续优化

如果事情只停留在“并行性差”和“状态流量大”，那 GDN 还只是难做快。

真正让它变成一道硬题的，是第三个约束：**数值稳定性。**

这次数据构造脚本里有一个特别关键的现象：

- 如果 `k` 直接来自 `torch.randn`
- 那么 `||k||^2` 的量级会接近 128
- state 在 delta rule 更新里可能每步被放大约 128 倍
- 从零状态开始，几十步后连 `float32` 都可能 overflow

所以脚本里必须做一件事：

**对 `k` 做 L2 normalize。**

这个细节非常说明问题。

它说明 GDN 的状态递推不是“公式写对了就行”，而是一个从输入尺度开始就要被严格控制的动态系统。否则你会遇到：

- state 爆炸
- 溢出
- 长程误差累积
- 低精度压缩后误差被不断放大

所以 GDN 的稳定性不是一个“后处理问题”，而是一个“性能优化前置条件”。

你如果不先把系统稳定住，后面所有的：

- FP16
- BF16
- FP8
- FP4

都很可能只是跑得飞快地错掉而已。

## 五、为什么论文里说它更线性，GPU 却不一定更喜欢它

很多人会有一个天然联想：

既然 GDN 这类结构不用显式构造 attention matrix，复杂度更线性，那它不是应该天然更适合硬件吗？

问题在于，**GPU 喜欢的从来不是“纸面复杂度更低”，而是“问题更像大块矩阵乘”。**

GPU 真正偏爱的，是这些东西：

- 足够厚的并行
- 足够高的算术强度
- 足够规则的访存
- 尽可能少的中间写回
- 尽可能高的 Tensor Core 利用率

而 GDN 偏偏在这些点上没有天然优势。

它当然理论上更线性，但它也同时带来了：

- 时间维递推依赖
- 高频 state 读写
- 低精度稳定性敏感

所以一个非常残酷、但也非常真实的事实是：

**论文里的渐进复杂度优势，不会自动翻译成 GPU 上的实际优势。**

中间还差一整套系统实现。

## 六、Decode 和 Prefill 看起来都是 GDN，实际上是两种完全不同的战场

### 1. Decode：这是单步状态更新的极限工程题

在 `gdn_decode_qk4_v8_d128_k_last` 里，一次只处理一个 token。核心计算更接近：

```text
[128 x 128] × [128]
```

也就是 mat-vec，而不是 mat-mat。

这件事直接决定了 decode 的优化方向：

- Tensor Core 用不上或者收益很有限
- state 流量是主矛盾
- launch overhead 特别敏感
- 小 batch 下 occupancy 很难好看

所以 decode 不是“怎么把算力榨满”的问题，而是：

**怎么把一次状态更新做到极致省流量、极致低开销。**

### 2. Prefill：这是一道“先把递推改造成并行”的算法工程题

prefill 不一样。它面对的是整段输入序列，而且还是 variable-length、packed layout 的场景。

如果你还按 token 一个个串行推过去，那它照样会：

- 并行度不足
- state 被反复读写
- 算术强度一直上不来

所以 prefill 的第一目标不是“再写一个更快 kernel”，而是：

**先把 recurrence 改写成 chunkwise recurrence 或 scan 风格问题。**

只有这样，chunk 内部的一部分计算才有机会矩阵化，才有机会靠近 Tensor Core 的甜点区。

所以如果一定要概括两者差别：

- decode 更像单步状态更新的工程极限
- prefill 更像把递推改写成块并行的算法极限

## 七、那 GDN 到底应该怎么优化？真正能落地的方案是什么

说了这么多，如果最后只剩一句“GDN 很难”，其实没有意义。真正关键的是：面对这些瓶颈，算子到底怎么做才有希望。

我会把落地方案拆成三条线。

### 第一条线：对 decode，目标是把单步状态更新做成极致 memory-efficient kernel

核心思路只有一句话：

**尽量一次读 state、在片上完成更新、尽量一次写回。**

具体可以做：

#### 1. 完整单步 fusion

把这些步骤尽量融合进一个 kernel：

- `g * S`
- `old_v = k @ S`
- `new_v` 门控混合
- `S += outer(k, delta)`
- `o = q @ S`

如果拆成多个 kernel，中间状态会在 HBM 上来回跑，launch overhead 也会被放大，基本等于主动放弃性能。

#### 2. 重排 state layout，保证 coalesced access

state 的布局不能拍脑袋定，要围绕线程映射和向量化 load/store 反推。

本质目标很明确：

- warp 访问连续
- transaction 尽量完整
- head / K / V 的映射和 kernel 的访存路径一致

#### 3. shared memory staging + swizzle

把 state tile 搬进 shared memory 做局部更新时，bank conflict 往往是隐形杀手。

对这类规则矩阵，做 SMEM swizzle 是非常值的一步，它不花哨，但很实用，尤其在小 batch 场景下经常是真正的收益点。

#### 4. persistent kernel / warp specialization

decode 天生碎，所以 launch overhead 会特别疼。

这时候 persistent kernel 或 warp specialization 都值得试，本质上是为了把“小步快跑”的代价降下来。

#### 5. 低精度存储 + 高精度累积

既然 decode 很多时候是 memory-bound，那么降低 state 字节数就是最直接的性能杠杆。

但不能盲降，合理路线通常是：

- state 用 FP16 / FP8 / FP4 存
- 更新和关键累积仍保留 FP32
- head 或 tile 级别维护 scale

这是最有希望同时兼顾速度和正确性的方向。

### 第二条线：对 prefill，目标是把递推改造成块并行

prefill 的关键不是单步做多快，而是把问题改写到 GPU 喜欢的形状。

#### 1. chunkwise recurrence

把长序列切成 chunk，例如 64 或 128 个 token 一块。

收益有两个：

- 一份 state tile 能在 chunk 内复用更久
- 一些 mat-vec 计算能被提升成 mat-mat

这一步几乎是 prefill 能不能做快的前提。

#### 2. chunk 内矩阵化，尽量吃 Tensor Core

GDN prefill 真正能享受 Tensor Core 红利的地方，不是在单 token 递推，而是在 chunk 内把多个 token 合起来处理。

也就是说：

- 块内并行
- 块间递推

这是比较现实的工程平衡点。

#### 3. length-aware 调度

prefill 一旦遇到 varlen，负载均衡就会成为大问题。

所以必须认真处理：

- 按长度分桶
- 减少 tail padding
- 减少 packed layout 下的分支发散

这些东西看起来不像 Tensor Core 那样“酷”，但常常决定最终是不是稳定快。

#### 4. 最小化块间 state 传递

chunking 如果每个 chunk 都完整回写 state，再完整读回，那收益会被抹掉很多。

所以真正好的 prefill 实现，不只是“有 chunk”，而是：

**让 state 在块内尽量多活一会儿，在块间尽量少搬一点。**

### 第三条线：对稳定性，目标是让低精度优化有资格成立

这是最容易被忽略、但最不能省的一条线。

#### 1. 对 `k` 做 normalize 或等价尺度控制

别让 state update 从第一步就带着错误量级起跑。

#### 2. 对 gate 做稳定参数化

通过 `sigmoid`、`softplus`、`exp(-exp())` 这类形式，把动态范围控制在可递推执行的区间里。

#### 3. 低精度存储，高精度累积

这是目前最现实的工程折中。

#### 4. 引入 tile/head 级别的分块 scale

如果要上 FP8/FP4，这一步基本绕不开，否则不同位置动态范围差异会让量化误差变得不可控。

## 八、真正的结论其实很朴素：GDN 不是算得慢，而是状态太重

如果一定要把全文压缩成最短的一句话，我会这么说：

**GDN 的核心矛盾，不是 FLOPs 不够低，而是状态太重。**

它重在三个地方：

- 递推依赖让并行变差
- 状态流量让带宽吃紧
- 低精度下又容易不稳定

这就是为什么 GDN 会给人一种很强烈的“理论很美，工程很硬”的感觉。

它当然不是没有潜力。恰恰相反，它很可能在长序列上非常有潜力。但前提是你得先解决一个 attention 世界里没那么中心的问题：

**怎么把一个状态系统，在 GPU 上做得既快、又稳、还不浪费带宽。**

所以最后我只留一句判断：

**GDN 最需要优化的不是乘法，而是状态。谁先把状态这件事做对，谁才真正拿到了这条路线的工程门票。**
